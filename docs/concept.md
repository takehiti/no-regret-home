# コンセプト - no-regret-home

---

## はじめに：「家は二度建てないと分からない」問題

建築業界では古くから、こう言われています。

> 「家は三度建てないと理想の家にならない」
> 「一度目は失敗、二度目でようやく分かる」

しかし現実には、多くの人にとって **家を建てるチャンスは一生に一度** です。

そして完成後、こう後悔します。

- 「もう少し天井を高くすればよかった」
- 「隣家からの視線がここまで気になるとは思わなかった」
- 「この位置に窓を作れば良かった」
- 「設計者に『それは無理です』と言われたが、なぜ無理なのか結局分からなかった」

**これは、本当に避けられないことなのでしょうか？**

---

## 問題の本質

### 1. **専門知識の壁**
建築には、構造・法規・設備・デザインなど多くの専門分野があります。
一般の建築主がこれらを理解し、判断することは困難です。

### 2. **想像と現実のギャップ**
図面や3Dパースを見ても、完成後の実際の体験は想像しにくいものです。
特に以下は図面では分かりません。

- 視線（どこから見えるか／見られるか）
- 圧迫感（天井高・空間の広がり感）
- 光と影（季節・時刻による変化）
- 音の伝わり方

### 3. **説明責任の欠如**
設計者・施工者から「それは無理です」「これが標準です」と言われたとき、
建築主はその理由を納得できる形で説明されることは稀です。

数値で示されても、それが自分の生活にどう影響するかは分かりません。

### 4. **意思決定の不可逆性**
一度建ててしまえば、やり直すことは極めて困難です。
数百万〜数千万円のコストと、時間的制約の中で、
建築主は「よく分からないまま」決断を迫られます。

---

## 解決のアプローチ

no-regret-home は、以下の方針でこの問題に取り組みます。

### 1. **数値と体験の橋渡し**
「天井高2.4m」と言われても、ピンと来ない人がほとんどです。
しかし **「この視点から見た時の見え方」** を示せば、誰でも判断できます。

→ **「判断できるビュー」を生成する**

### 2. **判定の根拠を可視化**
「それは法規上無理です」ではなく、
**「この斜線制限により、この高さまでしか建てられません」** を図で示す。

→ **「なぜ無理か」を説明できる資料を生成する**

### 3. **複数案の比較**
「フェンスを1.8mにした場合」と「2.0mにした場合」の違いを
視覚的に比較できるようにする。

→ **「どう変わるか」を体感できる**

### 4. **専門家の検証を支援**
設計者・施工者が建築主に説明する際のツールとしても使える。
プロ向けには Revit アドインとして高精度な検証機能を提供。

→ **「説明責任を果たす」ための道具**

## プロダクトの設計思想

### 思想1: **エンジンを核に据える**
ビジュアルや体験は重要ですが、その根幹には **正確な判定ロジック** が必要です。

- 視線が通るか／遮れるか
- 高さが十分か／圧迫感があるか
- 法規に適合するか

これらの判定を行う **UI非依存のエンジン** を中心に据えます。

エンジンが正しければ、あらゆるフロントエンド（Web / Unity / Revit）から
同じ判定結果を得られます。

### 思想2: **ビューは「判断」のため**
3Dモデルや VR は「体験」のためのツールとして有用ですが、
建築主にとって最も重要なのは **「これで良いか判断できること」** です。

- 鑑賞用の美しい3D < 判断できる代表視点の静止画
- 自由に歩き回れる3D < 「ここから見たらどうか」が分かる図

まずは **判断に必要な最小限のビュー** を提供します。
没入型体験は、その後のステップです。

### 思想3: **Revitは「プロ用検証ツール」**
一般の建築主は Revit を持っていません。
Revit アドインは以下の目的で位置づけます。

- 設計者が建築主に説明するための検証ツール
- 高精度な事例作成・検証
- エンジンの正確性を確認するためのテストベンチ

**Revit依存のロジックは作らない。**
エンジンが中心であり、Revitは「エンジンを使う例」の一つです。

### 思想4: **言語中立性**
エンジンは、特定のプログラミング言語に依存しない設計を目指します。

- TypeScript で実装しても、他言語に移植可能な設計
- データ構造と計算ロジックの明確な分離
- プラットフォーム固有のAPIに依存しない

これにより、将来的に以下が可能になります。

- C# での再実装（Revit / Unity）
- Python での実装（データ分析・バッチ処理）
- WebAssembly によるブラウザ実行

## ターゲットユーザー

### プライマリ: **建築主（施主）**
- 自分の家を建てる、または建てようとしている人
- 専門知識はないが、納得して決めたいと思っている
- 「それは無理です」に対して、理由を知りたい

### セカンダリ: **設計者・建築士**
- 建築主に対して、視覚的に分かりやすい説明をしたい
- 複数案を比較して提案したい
- 法規・構造の制約を可視化したい

### ターシャリ: **施工者・営業**
- 建築主の要望が実現可能かを判定したい
- 説明資料を自動生成したい

## 提供価値

### 建築主にとって
- ✅ **後悔を減らす** - 完成前に判断材料を得られる
- ✅ **納得して決める** - 理由が分かれば、諦めも決断もできる
- ✅ **説明できる** - 家族や第三者に判断根拠を共有できる

### 設計者にとって
- ✅ **説明コスト削減** - 視覚的資料で素早く伝わる
- ✅ **信頼構築** - 透明性のある説明で信頼を得る
- ✅ **検証効率化** - 手作業での確認が自動化される

### プロジェクト全体にとって
- ✅ **意思決定の質向上** - 曖昧さが減り、後戻りが減る
- ✅ **トラブル予防** - 完成後の「こんなはずじゃなかった」を防ぐ

## なぜ今、このツールが必要か

### 1. **技術の成熟**
- 3Dエンジン（Unity / Three.js）が普及
- BIM（Revit）が設計の標準に
- Web技術で複雑な可視化が可能に

### 2. **建築主の意識変化**
- 情報収集能力の向上
- 「プロに任せれば安心」から「自分で判断したい」へ
- SNS等で「後悔事例」が共有される時代

### 3. **説明責任の重要性**
- 単に「無理です」では通らない
- 透明性・納得性が求められる

## 最初の一歩: 視線フェンス判定

なぜ「フェンス」から始めるのか？

### 理由1: 誰でも理解できる
「隣から見えるか／見えないか」は、専門知識なしで判断できます。
しかし、実際には高さ・距離・地盤差の計算が必要です。

### 理由2: 典型的な後悔事例
「フェンスを作ったのに、2階から丸見えだった」
「もう少し高くすれば良かった」
このような後悔は非常に多い。

### 理由3: 計算ロジックがシンプル
視線判定は、幾何計算として明確です。
複雑すぎず、かといって自明でもない、良い題材です。

### 理由4: ビュー生成の練習
「どこから見えるか」を示す図は、他の判定（日射・圧迫感）にも応用できます。

## まとめ

**「家は二度建てないと分からない」を、二度建てずに実現する。**

これが no-regret-home のミッションです。

- 判断できるビューを提供する
- 理由を可視化する
- 納得して決められる状態を作る

一般の建築主が、後悔しない家づくりを一度で実現できる未来へ。
